# ADR-0017: CLI Package and Binary Invocation Surface

## Status

Proposed

## Context

The Multiverse CLI currently lives at `apps/cli/src/index.ts` and is invoked in two ways:

- **Development shortcut**: `pnpm cli ...` — runs `tsx apps/cli/src/index.ts` directly, using TypeScript source with the `tsx` runtime. This is a repo-local developer convenience, not a stable external interface.
- **No formal binary**: there is no compiled, installable, or linkable `multiverse` binary. Users cannot run `multiverse derive` unless they clone the repo and use the pnpm shortcut.

This is appropriate for early development but becomes a gap as the tool matures. The CLI surface needs a formal, compiled invocation path that:

1. Does not require `tsx`, the monorepo structure, or pnpm to be on the PATH at runtime.
2. Can be linked or installed as a real CLI tool.
3. Provides a stable surface for documentation, demos, and external use.

The decision establishes how `apps/cli` transitions from a development-only package to a package with a formal binary.

## Decision

### 1. `apps/cli` is formalized as a CLI package that produces a `multiverse` binary

`apps/cli/package.json` gains:
- A `"bin"` entry: `{ "multiverse": "./bin/multiverse.js" }`
- A `"build"` script that compiles `src/index.ts` to `dist/`
- A `"main"` field pointing to `./dist/index.js`

The package remains `private: true`. A public registry publish is out of scope for 1.0.

### 2. The formal binary is compiled JavaScript, not tsx-sourced TypeScript

The `multiverse` binary is the output of `tsc`, not a `tsx`-wrapped TypeScript invocation.

`tsx` remains a dev dependency. It powers `pnpm cli` for fast in-repo iteration. It must not appear in the runtime path of the compiled binary.

Rationale: a binary that requires `tsx` at runtime would silently break on any machine where `tsx` is not installed. The compiled output is self-contained Node.js JavaScript that requires only a compatible `node` version.

### 3. A thin shebang entry point wraps the compiled output

The `bin/multiverse.js` file is a hand-authored, version-controlled wrapper:

```js
#!/usr/bin/env node
import('../dist/index.js').catch(console.error);
```

This file:
- carries the `#!/usr/bin/env node` shebang required by npm/pnpm for bin scripts
- is committed to the repository as a static file (not generated by tsc)
- has no logic of its own; it only bootstraps the compiled entry point
- must be `chmod +x` in the repository

The compiled output (`dist/index.js`) is excluded from version control (`.gitignore`). A `build` step is required before the binary is usable.

### 4. The compilation target is straightforward `tsc` output — no bundling

The build uses `tsc` with a dedicated `tsconfig.build.json` inside `apps/cli`:

- Input: `src/index.ts`
- Output directory: `dist/`
- Module format: `ESNext` (consistent with the rest of the workspace)
- Source maps: generated for debuggability
- Declaration files: not required (private package, no consumers of the type surface)

Bundling (esbuild, rollup, etc.) is explicitly deferred. The compiled output will reference workspace packages as bare imports. This means the binary is usable only within the workspace (or with proper `node_modules` resolution), which is acceptable for the initial implementation.

### 5. `pnpm cli` remains the repo-development shortcut — unchanged

The root `package.json` `cli` script (`tsx apps/cli/src/index.ts`) is not modified. It continues to serve as the fast, no-build-required dev path.

The distinction is:

| Invocation | Path | Requires build? | Requires tsx? |
|------------|------|-----------------|---------------|
| `pnpm cli ...` | TypeScript source via tsx | No | Yes |
| `multiverse ...` | Compiled JS via node | Yes | No |

Both paths invoke the same `runCli` function. The source of truth for behavior is the TypeScript implementation, not which path was used to invoke it.

### 6. Local / workspace-linked usage is sufficient for the initial implementation

The first implementation slice does not define a global install or registry publish story.

Acceptable initial usage patterns:

- `pnpm --filter @multiverse/cli build` to compile
- `node apps/cli/bin/multiverse.js ...` directly
- `pnpm link` (or `npm link`) within the workspace to expose `multiverse` on PATH locally

A polished global install story (npm publish, Homebrew tap, standalone binary) is a future concern and out of scope until the tool is ready for broader distribution.

### 7. The build is integrated into the workspace but not forced into the test pipeline

The CLI build (`pnpm build`) is added to the workspace-level scripts. It is not a required prerequisite for `pnpm test`, since tests invoke `runCli` programmatically through the TypeScript source path.

The CI pipeline may optionally run `pnpm build` as a compile check to catch build-specific errors (import resolution, output shape), but test runs are not gated on it.

### 8. The `src/index.ts` isMainModule guard must remain correct after compilation

The current entry point guards `main()` with:

```ts
const isMainModule =
  process.argv[1] !== undefined &&
  import.meta.url === new URL(`file://${process.argv[1]}`).href;

if (isMainModule) {
  await main();
}
```

After compilation, `import.meta.url` in the output is the URL of `dist/index.js`, and `process.argv[1]` when invoked via `bin/multiverse.js` points to the bin wrapper. The guard will evaluate to `false` in that case.

To ensure `main()` is called when the binary is invoked, the `bin/multiverse.js` wrapper must call `main` explicitly, or `src/index.ts` must export `main` and the compiled entry must call it unconditionally.

**Resolved approach**: export `main` from `src/index.ts` and call it from `bin/multiverse.js` after import, instead of relying on the isMainModule guard in the binary path. The isMainModule guard is retained for the `pnpm cli` path (direct tsx invocation).

## Consequences

### Positive

- `multiverse` becomes a real binary, not a pnpm shortcut
- the compiled binary does not require `tsx`, `ts-node`, or the TypeScript toolchain at runtime
- `pnpm cli` continues to work unchanged for in-repo development
- no bundler complexity is introduced
- the `runCli` function surface is unchanged; tests are unaffected

### Negative

- the compiled binary requires workspace `node_modules` to resolve package imports (not standalone)
- a build step is now required before the binary is usable; this is not automatic
- the `bin/multiverse.js` → `dist/index.js` indirection adds a small layer that must be kept in sync

## Open questions before implementation

### Q1: `main()` export and bin wrapper calling convention

Should `bin/multiverse.js` call `main()` directly after import, or should it rely on a top-level `await main()` in the compiled output?

Proposed: export `main` from `src/index.ts`; `bin/multiverse.js` imports and awaits it. The isMainModule guard then only serves the `pnpm cli` path.

### Q2: `apps/cli` tsconfig relationship to root tsconfig

The existing root `tsconfig.json` includes `apps/**/*.ts`. The CLI build tsconfig should either extend the root or be self-contained. Extending root avoids drift; self-contained avoids coupling.

Proposed: `apps/cli/tsconfig.build.json` extends the root `tsconfig.json`, overrides `outDir`, `rootDir`, and disables `declaration`.

### Q3: CI build check

Should the CI pipeline run `pnpm build` as a compile-check step (separate from typecheck)?

Proposed: add a `build` step to CI that runs `pnpm --filter @multiverse/cli build` to catch output-specific failures (e.g. missing `dist/` references). This is a check, not a gate on tests.
